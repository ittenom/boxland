Tasks 45-62: asset pipeline + Asset Manager UI + pixel editor + waveform

Designers can now upload sprite/tile/audio assets, edit metadata via the
generic form renderer, and the publish pipeline applies drafts atomically.

  Schema (45-49):
    - assets (single canonical table; kind discriminator + metadata_json)
    - asset_animations (named tags within a sheet)
    - palettes (with default retro palette seeded)
    - palette_variants (recipes, source_to_dest_json)
    - asset_variants (baked outputs; status pending/baked/failed)

  Upload + parsers (50-55):
    - Upload endpoint: multipart, sniff-based content-type validation,
      content-addressed paths via sha256, idempotent dedup
    - Importer registry with auto-detect + explicit-override dropdown
    - 5 parsers: raw PNG manual-grid, generic strip (h/v/rows/cols),
      Aseprite JSON sidecar (hash + array variants), TexturePacker JSON
      (with filename-pattern animation inference), free-tex-packer JSON
    - 32 unit + integration tests across the parser surface

  Image normalization + audio (56-57):
    - 32x32 cell validation with structured warnings (non-square,
      non-canonical, mixed sizes, empty sheet)
    - WAV duration extraction in pure Go (RIFF chunk walker, no deps)
    - OGG/MP3 detected via magic bytes; duration deferred (placeholder
      metadata until later refinement)

  Bake job (58):
    - Per-pixel color remap via NRGBA pixel pass
    - Output keyed by sha256(source_path || sorted recipe entries) so
      identical (source, recipe) inputs always land at the same path ->
      idempotent across re-runs (Reused=true in result)
    - Multiple recipes per asset bake in parallel via sync.WaitGroup
    - Baked variants written to bytes.NewReader (seekable) so AWS SDK v2
      stops complaining about MinIO PutObject without TLS+trailing-checksum

  Asset as Artifact[T] (59):
    - First concrete user of the publishing/artifact framework
    - AssetDraft (name, tags) implements configurable.Configurable so the
      generic form renderer drives the editor UI
    - Handler.Publish reads previous values, applies update inside the
      pipeline tx, computes structured diff for the publish_diffs row
    - Wire-up: cmd/boxland registers the handler in publishRegistry

  Asset Manager UI (60):
    - assets.templ: list page with grid + tag/kind filter + search
      (HTMX-driven; #assets-grid swaps in place on every input)
    - asset_upload.templ: drag-drop modal with kind override
    - asset_detail.templ: per-asset modal with the generic form renderer
      driving the AssetDraft editor + delete button
    - 7 server handlers (list, grid, upload-modal, upload, detail, delete,
      draft) wired under /design/assets/*
    - 5 integration tests covering the full HTML response shapes
    - data-bx-action="open-upload" wired in boot.js so the toolbar button
      fetches the modal via HTMX
    - Asset card + drop zone + detail-pane CSS in pixel.css

  Canvas2D pixel editor (61):
    - PixelBuffer (ImageData wrapper, get/set/blit/toPNGBlob)
    - PixelEditor binds tools to the shared CommandBus so undo/redo +
      hotkey rebinding work uniformly
    - Tools: pencil (B), eraser (E), color picker (I), undo (Mod+Z),
      redo (Mod+Shift+Z)
    - Stroke commit captures every changed pixel as one undo entry
    - POST /design/assets/{id}/replace creates a NEW asset (immutable
      content-addressed path; original kept) and triggers re-bake of any
      palette variants the new asset already has
    - 11 TS tests; deliberately uses a small jsdom shim for ImageData

  Audio waveform (62):
    - mountWaveform decodes via Web Audio API (handles WAV/OGG/MP3 in any
      modern browser), downsamples to per-bin peak amplitudes, paints
      chunky pixel bars in the brand accent color
    - Falls back to a hairline placeholder on fetch/decode failure so the
      UI never renders an empty box
    - Waveform + native <audio controls loop preload=metadata> in
      asset_detail.templ; 4 TS tests cover happy/fail/decode-fail paths

Test totals: 13 Go packages, 116 TS tests across 15 suites, all green.

Justfile: test-go now passes -p 1 so test packages run sequentially. The
integration tests share a single dev Postgres; parallel package execution
clobbered each other's data. Within-package parallelism is unaffected.
