Tasks 98-108: maps + WS-driven Mapmaker authoring

Map authoring now flows through the SAME WS gateway players use, per
PLAN.md §4j unified-broadcaster: a designer painting tiles in Mapmaker
emits DesignerCommand::PlaceTiles -> dispatcher writes through
maps.Service -> MarkChunksDirty bumps the spatial.Grid chunk version ->
the next broadcaster tick pushes a Diff to every subscriber on the
map's chunks (designers + players, one pipeline).

Decisions worth flagging:
  * spatial.Grid gained BumpVersion(c) -- the right primitive for tile-
    layer mutations that don't move any moving-entity. Without it the
    grid would never see lighting / tile edits and AOI broadcast would
    skip them.
  * runtime.InstanceManager uses a per-key build barrier (channel) so
    two simultaneous JoinMaps for the same instance only pay the
    Recover() cost once.
  * Authoring opcodes (200..) ride the existing DesignerCommand verb so
    the gateway's realm-violation check covers them without bespoke
    per-opcode auth. Player-realm conns sending DesignerCommand still
    get a PolicyViolation close.
  * canonicalInstanceID() is "live:{map_id}:0" until per-user / per-
    party routing lands. Sandbox uses "sandbox:{designer_id}:{map_id}"
    in a separate id space, also routed through this same path.
  * The Mapmaker create-modal uses a custom small form rather than the
    generic MapDraft.Descriptor() form because the create path needs
    width + height which aren't editable post-create (hence not in
    Descriptor). All other map-edit surfaces use the generic renderer.
  * WS authoring tests guard the realm boundary: a player-realm
    handshake with a fake JWT closes with PolicyViolation BEFORE the
    DesignerCommand check ever runs (auth-handshake-time enforcement).
